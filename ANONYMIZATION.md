# Анонимизация данных в 1C MCP Toolkit

Функционал анонимизации позволяет автоматически маскировать персональные и конфиденциальные данные, которые 1С возвращает AI-агенту. Агент работает с токенами вида `[ORG-00001]` вместо реальных наименований, ИНН, телефонов и т.д., при этом может передавать эти токены обратно в запросы — сервер сам подставит реальные значения перед исполнением.

---

## Как работает анонимизация (общее)

### Принцип токенизации

Каждое уникальное чувствительное значение заменяется стабильным токеном:

```
"ООО «Газпром»"        →  [ORG-00001]
"7707083893"           →  [INN-00001]
"+7 (999) 123-45-67"   →  [PHONE-00001]
"Иванов Иван Иванович" →  [PER-00001]
```

Токен детерминированный: одно и то же значение всегда получает один и тот же токен в рамках сессии. Агент может использовать токен в параметрах следующего запроса — система автоматически восстановит реальное значение перед отправкой в 1С.

### Категории токенов

| Токен | Описание | Пример значения |
|-------|----------|-----------------|
| `[ORG-NNNNN]` | Организации, контрагенты | ООО «Ромашка» |
| `[PER-NNNNN]` | Физические лица, ФИО | Иванов Иван Иванович |
| `[INN-NNNNN]` | ИНН (10 или 12 цифр) | 7707083893 |
| `[KPP-NNNNN]` | КПП (9 цифр) | 770701001 |
| `[OGRN-NNNNN]` | ОГРН (13 цифр) | 1027700132195 |
| `[SNILS-NNNNN]` | СНИЛС | 123-456-789 01 |
| `[PHONE-NNNNN]` | Телефон | +7 (999) 123-45-67 |
| `[EMAIL-NNNNN]` | Электронная почта | user@example.com |
| `[PASSPORT-NNNNN]` | Паспортные данные | — |
| `[BIK-NNNNN]` | БИК банка | 044525225 |
| `[ACC-NNNNN]` | Расчётный счёт (20 цифр) | 40702810900000000001 |
| `[SWIFT-NNNNN]` | SWIFT/BIC код | GAZPRUMM |
| `[IBAN-NNNNN]` | IBAN | RU12 3456 7890 … |
| `[LOC-NNNNN]` | Адреса, города, регионы | г. Москва, ул. Ленина |
| `[DATE-NNNNN]` | Даты | 01.01.2024 |
| `[STR-NNNNN]` | Произвольные строки | Текст комментария |
| `[CARD-NNNNN]` | Номера банковских карт (PAN, проверка Луна) | 4111 1111 1111 1111 |

## Раздел 1: Анонимизация в режиме «Встроенный сервер»

В этом режиме вся логика анонимизации выполняется непосредственно в 1С — на стороне обработки, без Python и без прокси.

### Как включить

В обработке перейдите на вкладку **«Анонимизация»** и установите флаг **«Использовать анонимизацию по правилам»**.

### Как работает

Анонимизация применяется к результатам инструмента `execute_query`. Обработка строится на двух механизмах: **политики полей** и **регулярные выражения**.

#### Политики полей

Каждая колонка результата обрабатывается согласно своей политике, заданной в настройках:

- **`force_anonymize`** — поле помечено как «Анонимизировать: Да». Значение целиком токенизируется. Категория берётся из явно заданного правила, либо определяется автоматически по имени/пути поля (см. раздел «Автоопределение категории»).

- **`default`** — поле не помечено явно. К значению применяется **словарный поиск**: все подстроки, совпадающие с предзагруженными из базы 1С значениями (с проверкой границ слова), заменяются токенами.

- **`force_keep`** — поле помечено как «Анонимизировать: Нет». Значение не трогается.

Словарь загружается из базы 1С в момент **запуска встроенного сервера** (кнопка «Запустить сервер»), а также при включении флага «Использовать анонимизацию по правилам» на уже работающем сервере. Система запрашивает значения из полей, помеченных флагом «В словарь» — до 100 000 уникальных значений на поле.

#### Регулярные выражения (опционально)

После обработки политик применяется фаза регулярных выражений — если они настроены. Паттерны применяются к текстам после словарной обработки, совпадения токенизируются с указанной категорией.

### Настройка правил

Откройте форму через кнопку **«Правила анонимизации...»** на вкладке «Анонимизация». Форма называется **«Настройка правил анонимизации»** и состоит из двух частей.

#### Дерево правил для полей

Отображает иерархию объектов метаданных текущей базы (колонка **«Объект»**). Для каждого поля можно задать:

| Колонка | Описание |
|---------|----------|
| **Анонимизировать** | «Да» / «Нет» для данного поля |
| **В словарь** | Загружать значения этого поля в словарь замен |
| **Категория** | Явная категория токена (ORG, PER, INN и т.д.) |

Поля с **«Анонимизировать: Да»** токенизируются принудительно. Поля с флагом **«В словарь»** участвуют в словарном поиске для прочих полей (режим `default`).

Фильтры над деревом: **«Отмеченные»** — показать только отмеченные поля; **«В словаре»** — только поля с флагом «В словарь»; переключатель **«Анонимизировать: Все / Да / Нет / Не задано»**.

#### Агрессивная анонимизация

Флаг **«Агрессивная анонимизация»** (вверху формы) заставляет систему токенизировать все строковые поля и представления ссылок, не отмеченные явно как «Нет». Используйте для максимальной защиты данных.

#### Раздел «Regex-правила анонимизации»

Сворачиваемый раздел в нижней части формы. Содержит таблицу дополнительных паттернов:

| Колонка | Описание |
|---------|----------|
| **Вкл** | Использовать правило |
| **Шаблон (PCRE2)** | Регулярное выражение |
| **Категория** | Категория токена для совпадений |
| **Описание** | Комментарий |

### Автоопределение категории по имени поля

Для полей без явно заданной категории система автоматически определяет её по имени поля. Точное соответствие или вхождение подстроки:

| Подстрока в имени поля | Автокатегория |
|------------------------|---------------|
| `ИНН` / `INN` | INN |
| `КПП` / `KPP` | KPP |
| `ОГРН` / `OGRN` | OGRN |
| `СНИЛС` / `SNILS` | SNILS |
| `БИК` / `BIK` | BIK |
| `СЧЕТ` / `ACCOUNT` | ACC |
| `ТЕЛЕФОН` / `PHONE` | PHONE |
| `EMAIL` / `ПОЧТА` | EMAIL |
| `ПАСПОРТ` / `PASSPORT` | PASSPORT |
| `ДАТА` / `DATE` | DATE |
| `АДРЕС` / `ADDRESS`, `СТРАНА`, `ГОРОД`, `УЛИЦА`, `ДОМ`, `КВАРТИРА` | LOC |
| `КОНТРАГЕНТ`, `ОРГАНИЗАЦИЯ`, `PARTNER` | ORG |
| `ФИЗЛИЦО`, `СОТРУДНИК`, `PERSON`, `АВТОР`, `ОТВЕТСТВЕННЫЙ`, `ФИО`, `ФАМИЛИЯ` | PER |

### Ссылочные типы

Ссылочные значения в результате `execute_query` имеют вид объекта с полем `Представление`:

```json
{
  "_objectRef": true,
  "ТипОбъекта": "Справочник.Контрагенты",
  "Представление": "ООО Ромашка"
}
```

Анонимизируется только поле **`Представление`** — строковое отображение ссылки. Технические поля (`_objectRef`, `ТипОбъекта`) остаются нетронутыми.

#### Настройка «Наименование» распространяется на все ссылочные поля

Это ключевой момент. В схеме результата каждая колонка содержит массив `sources` — пути к исходным полям. Например, колонка `КонтрагентСсылка` в схеме имеет `sources: ["Контрагент.Ссылка.Наименование"]`.

При обработке система извлекает последний сегмент пути (`Наименование`) и использует его как подсказку. Если вы настроили правило **«Анонимизировать: Да»** для поля `Справочник.Контрагенты.Наименование`, то **все ссылочные колонки**, чьи `sources` указывают на это поле, также будут анонимизированы — даже если имя самой колонки нейтральное.

Обратное тоже верно: **«Анонимизировать: Нет»** для `Наименование` → представления всех ссылок на этот справочник сохранятся.

### Что никогда не анонимизируется

- **Числа, булевы значения, `null`** — анонимизируются только строки; нечисловые типы не затрагиваются в принципе
- Поля с явной политикой **«Анонимизировать: Нет»** в правилах
- Значения полей-перечислений (`ПеречислениеСсылка.*`) — даже при `force_anonymize` или агрессивном режиме

### Что по умолчанию не анонимизируется (но допускает явный override)

- Значения полей типа ПланыСчетов (`ПланСчетовСсылка.*`) — **по умолчанию не анонимизируются**, в том числе при агрессивном режиме (план счетов содержит нормативные данные: коды и наименования счетов, без персональных данных). Можно переопределить через точные правила анонимизации (`force_anonymize`).

### Хранение настроек

Правила сохраняются в `ХранилищеОбщихНастроек` текущей базы 1С в двух независимых ключах:
- правила полей — `MCPToolkit / ExecuteQueryFieldRules.v2`
- regex-правила — `ExecuteQueryRegexRules.v1 / rules`

**Экспорт** — только в JSON. **Импорт** — поддерживает JSON и XML-формат выгрузки типовой обработки 1С «Скрытие конфиденциальной информации».

## Раздел 2: Анонимизация в режиме «Прокси»

В этом режиме анонимизация выполняется на стороне Python-прокси: перехватывает ответы 1С и применяет токенизацию до того, как результат достигнет AI-агента.

### Как включить

Передайте переменную окружения при запуске:

```bash
# Docker run
docker run -e ANONYMIZATION_ENABLED=true ...

# Docker Compose
environment:
  - ANONYMIZATION_ENABLED=true

# Прямой запуск
ANONYMIZATION_ENABLED=true python -m onec_mcp_toolkit_proxy
```

По умолчанию анонимизация **выключена** (`ANONYMIZATION_ENABLED=false`).

### Переменные окружения

#### Основные параметры

| Переменная | Умолчание | Описание |
|-----------|-----------|----------|
| `ANONYMIZATION_ENABLED` | `false` | Включить/выключить анонимизацию |
| `ANONYMIZATION_TOOLS` | `execute_query,execute_code,`<br>`get_object_by_link,`<br>`get_event_log,`<br>`find_references_to_object` | Инструменты, к которым применяется анонимизация |
| `ANONYMIZATION_RADICAL_MODE` | `false` | Токенизировать все строки (без анализа имён полей) |
| `ANONYMIZATION_INCLUDE_ERRORS` | `true` | Анонимизировать тексты ошибок |

#### SpaCy NER

| Переменная | Умолчание | Описание |
|-----------|-----------|----------|
| `ANONYMIZATION_NER_ENABLED` | `false` | Включить нейросетевое распознавание NER |
| `ANONYMIZATION_NER_FOR_EXECUTE_CODE_RESULT` | `false` | NER для результатов `execute_code` |
| `ANONYMIZATION_NER_FOR_ERROR` | `false` | NER внутри текстов ошибок |
| `ANONYMIZATION_SPACY_MODEL` | `ru_core_news_md` | Имя модели SpaCy (должна быть установлена в окружении) |

> NER работает медленнее регулярных выражений. Рекомендуется оставить выключенным, если не нужно обнаружение сущностей в произвольном тексте (комментарии, описания).

#### Словарь

| Переменная | Умолчание | Описание |
|-----------|-----------|----------|
| `ANONYMIZATION_DICTIONARY_ENABLED` | `true` | Включить словарный поиск |
| `ANONYMIZATION_DICTIONARY_SOURCES_OVERRIDE` | — | JSON-список источников (полная замена стандартного) |
| `ANONYMIZATION_DICTIONARY_SOURCES_ADD` | — | JSON-список источников (добавление к стандартному) |
| `ANONYMIZATION_DICTIONARY_APPLY_KEY_SUBSTRINGS_OVERRIDE` | — | CSV: подстроки имён полей, к которым применяется словарь (замена) |
| `ANONYMIZATION_DICTIONARY_APPLY_KEY_SUBSTRINGS_ADD` | — | CSV: добавить подстроки к стандартным |

**Стандартные источники словаря:**

| Источник | Категория | Дополнительные поля |
|----------|-----------|---------------------|
| Справочник.Контрагенты | ORG | НаименованиеПолное |
| Справочник.Организации | ORG | НаименованиеПолное |
| Справочник.ФизическиеЛица | PER | Фамилия, Имя, Отчество, НаименованиеСлужебное |
| Справочник.Сотрудники | PER | — |
| Справочник.Пользователи | PER | — |

Словарь применяется к полям, в имени которых есть: `наименован`, `представлен`, `_presentation`, `note`.

**Пример: добавить справочник в словарь**

```bash
ANONYMIZATION_DICTIONARY_SOURCES_ADD='[{"from": "Справочник.Партнеры", "category": "ORG", "extra_fields": ["НаименованиеПолное"]}]'
```

#### Ключевые подстроки (тонкая настройка)

| Переменная | Описание |
|-----------|----------|
| `ANONYMIZATION_SENSITIVE_KEY_SUBSTRINGS` | JSON `{"подстрока": "КАТЕГОРИЯ"}` — полная замена стандартного списка |
| `ANONYMIZATION_SENSITIVE_KEY_SUBSTRINGS_ADD` | JSON — добавление к стандартному |
| `ANONYMIZATION_SENSITIVE_KEY_EXACT` | JSON точных совпадений (полная замена) |
| `ANONYMIZATION_SENSITIVE_KEY_EXACT_ADD` | JSON точных совпадений (дополнение) |
| `ANONYMIZATION_NON_SENSITIVE_KEY_SUBSTRINGS` | CSV — «нечувствительные» подстроки (полная замена) |
| `ANONYMIZATION_NON_SENSITIVE_KEY_SUBSTRINGS_ADD` | CSV — добавление |

**Пример: добавить поле «Ответственный» как персональные данные**

```bash
ANONYMIZATION_SENSITIVE_KEY_SUBSTRINGS_ADD='{"ответственный": "PER"}'
```

### Как работает прокси: поток обработки

```
Запрос от агента (например, execute_query)
         │
         ▼
  Де-токенизация параметров
  [ORG-00001] → «ООО Газпром»
         │
         ▼
  Запрос к 1С через long polling
  (схема запрашивается принудительно)
         │
         ▼
  Ответ от 1С (данные + схема)
         │
         ▼
  Анонимизация ответа
  (словарь → whole-value детекция → forced-name/row-group → inline замены)
         │
         ▼
  Удаление схемы (если агент не запрашивал)
         │
         ▼
  Ответ агенту с токенами
```

### Pipeline обнаружения (фактический порядок)

Для каждого строкового значения применяются последовательно:

1. **Словарь** — поиск подстрок из предзагруженного списка наименований; применяется только к полям с именами `наименован`, `представлен`, `_presentation`, `note`. Для `execute_code` словарь применяется ко всему `data`-поддереву целиком.
2. **Whole-value детекция** — определение категории для значения целиком:
   - Анализ имени поля (`контрагент` → ORG, `инн` → INN и т.д.) с учётом source hints из схемы
   - Regex full-match PATTERNS (весь текст должен соответствовать паттерну целиком)
   - SpaCy NER (если включён)
   - Детекция номеров банковских карт по алгоритму Луна (whole-string)
   - Радикальный режим — всё остальное как STR (если `ANONYMIZATION_RADICAL_MODE=true`)
3. **Forced-name / row-group эвристики** — принудительная токенизация полей `Наименование`, `Представление` на основе типа родительского объекта; контекстная группировка строк для определения ORG-полей
4. **Inline замены** — если whole-value детекция не дала результата, ищутся PII-фрагменты внутри строки через SEARCH_PATTERNS (regex **без привязки к началу/концу**), SpaCy NER и детекцию номеров карт по Луну

### Умная анонимизация псевдонимов: схема и source hints

Для `execute_query` прокси запрашивает у 1С расширенную схему результата. Схема содержит информацию об источниках каждой колонки:

```json
{
  "schema": {
    "columns": [
      {
        "name": "X1",
        "sources": ["Справочник.Контрагенты.Наименование"]
      }
    ]
  }
}
```

Из `sources` извлекается последний сегмент пути (`Наименование`), который используется как дополнительный ключ. Это позволяет корректно анонимизировать результаты с псевдонимами колонок:

```
// Колонка X1 — это Наименование контрагента
ВЫБРАТЬ Контрагенты.Наименование КАК X1
ИЗ Справочник.Контрагенты КАК Контрагенты
```

- **Без схемы:** `X1` не распознаётся → значение может не анонимизироваться
- **Со схемой:** `X1` → источник `Наименование` → активируется словарь → значение токенизируется

Схема запрашивается автоматически (агенту не нужно явно передавать `include_schema`).

### Ссылочные типы

Ссылочные значения в результате `execute_query` представлены объектом с полем `Представление`:

```json
{
  "_objectRef": true,
  "ТипОбъекта": "Справочник.Контрагенты",
  "Представление": "ООО Ромашка"
}
```

Анонимизируется только поле **`Представление`**. Технические поля `_objectRef` и `ТипОбъекта` остаются нетронутыми.

#### Настройка «Наименование» распространяется на все ссылочные поля

Это ключевой момент для понимания системы. В схеме результата каждая колонка содержит массив `sources` — пути к исходным полям в 1С. Прокси извлекает из них последний значимый сегмент и использует как подсказку при анонимизации.

Пример: колонка `Контрагент` содержит ссылочное значение. В схеме у неё `sources: ["Контрагент.Ссылка.Наименование"]`. Прокси извлекает `Наименование` и использует его как дополнительный ключ. Затем:

1. **Словарный поиск** активируется, если в ключах есть `наименован` — словарь заменит известные наименования токенами
2. **Forced-name логика** — поле `Наименование` входит в `FORCE_NAME_KEYS`, поэтому если `ТипОбъекта` объекта указывает на категорию (например, `Справочник.Контрагенты` → `ORG`), `Представление` будет принудительно токенизировано как `[ORG-NNNNN]`

Следствие: **достаточно настроить анонимизацию поля `Наименование` в справочнике** — и все ссылочные колонки, которые указывают на это поле через `sources`, будут анонимизированы автоматически. Настройка `force_keep` на `Наименование` аналогично заблокирует анонимизацию представлений ссылок на этот справочник.

То же самое работает для `НаименованиеПолное`, `Представление`, `_presentation`.

### Политики анонимизации для колонок execute_query

Схема может задавать явную политику для каждой колонки:

| Политика | Поведение |
|----------|-----------|
| `default` | Применять стандартные правила (словарь, source hints, force-name логика) |
| `force_keep` | Никогда не анонимизировать ни строку, ни представление ссылки |
| `force_anonymize` | Всегда токенизировать целиком |

Колонки, все типы которых — `ПеречислениеСсылка.*`, автоматически получают `force_keep`.

Колонки, все типы которых — `ПланСчетовСсылка.*`, также автоматически получают `force_keep` по умолчанию (планы счетов не содержат персональных данных). Исключение: если в точных правилах анонимизации для реквизита явно задан `force_anonymize` — он будет применён.

**Защита строковых субатрибутов плана счетов** (например, `СчетДт.Представление`, `СчетДт.Код`) работает в обоих режимах («Встроенный сервер» и «Прокси»): BSL анализирует `sources` колонки через метаданные 1С (`РазрешитьПутиРазыменования`) и устанавливает `anonymization_mode: "force_keep"` до применения агрессивной анонимизации.

### Контекстная анонимизация по группировке строк

При запросе, возвращающем данные по нескольким организациям, система распознаёт группы автоматически:

```
ВЫБРАТЬ
    Поставщик.Наименование КАК ИмяПоставщика,
    Поставщик.ИНН          КАК ИНН1,
    Покупатель.Наименование КАК ИмяПокупателя,
    Покупатель.ИНН          КАК ИНН2
ИЗ ...
```

Логика: `ИНН1` рядом с `ИмяПоставщика` → рядом есть признак ИНН → `ИмяПоставщика` принудительно получает категорию `ORG`. Аналогично для второй группы.

### Что никогда не анонимизируется

**Числа, булевы значения, `null`** — анонимизируются только строки. Любое нестроковое скалярное значение (int, float, bool, null) возвращается без изменений независимо от настроек.

**Служебные поля верхнего уровня ответа** — всегда пропускаются целиком:
`success`, `truncated`, `has_more`, `count`, `limit`, `returned`, `offset`, `next_offset`, `last_date`, `next_same_second_offset`, `schema`

**Технические поля на любом уровне** (`SKIP_KEYS`) — значение не трогается:
`_ref`, `_objectRef`, `_type`, `УникальныйИдентификатор`, `ТипОбъекта`, `session`, `level`, `event`, `application`, `transaction_status`

**Поля с «нечувствительными» именами** — подстрока в имени поля указывает, что значение является техническим, а не персональным:

| Подстрока | Примеры полей |
|-----------|---------------|
| `файл`, `модуль`, `шаблон`, `макет` | ИмяФайла, ИмяМодуля |
| `таблиц`, `колонк`, `измерени`, `ресурс`, `реквизит` | ИмяТаблицы, ИмяКолонки |
| `конфигурац`, `расширени`, `ключ`, `индекс`, `нумератор` | — |
| `количеств`, `metadata`, `config`, `template`, `column`, `field` | — |

**Поля, имя которых начинается на «статусный» префикс** — подавляется целочисленная токенизация:
`тип*`, `статус*`, `код*`, `режим*`, `вид*`, `форма*`, `категория*`, `способ*`, `результат*`, `признак*`, `роль*`, `состояние*`, `порядок*`, `формат*`, `уровень*`, `этап*`, `раздел*`, `важность*`, `метод*`, `вариант*`, `единица*`, `действие*`, `настройка*`, `причина*`, `приоритет*`, `степень*`, `характер*`, `цвет*`, `цель*`

**Значения-перечисления** (`ТипОбъекта` начинается с `ПеречислениеСсылка.`) — не анонимизируются даже при `force_anonymize`.

**Значения планов счетов** (`ТипОбъекта` начинается с `ПланСчетовСсылка.`) — по умолчанию не анонимизируются, в том числе при агрессивном режиме: автоматически получают `force_keep`, если `mode` и `reference_presentation_mode` колонки явно не заданы как `force_anonymize`.

**Заведомо безопасные значения** — не токенизируются никогда:
`Да`, `Нет`, `Истина`, `Ложь`, `True`, `False`, `Проведен`, `НеПроведен`, `Записан`, `Committed`, `RolledBack`, `NotApplicable`, `Unfinished`, `Information`, `Warning`, `Error`, `Note`, `USD`, `EUR`, `руб.`, `Евро`, `Доллар США`, `Российский рубль`

### Изоляция токенов по каналам

Каждый канал имеет независимый маппинг токенов:

```
channel=dev   →  [ORG-00001] = «ООО Ромашка»
channel=prod  →  [ORG-00001] = «АО МегаКорп»
```

### Примеры запуска

**Минимальный:**

```bash
docker run -d -p 6003:6003 \
  -e ANONYMIZATION_ENABLED=true \
  roctup/1c-mcp-toolkit-proxy
```

**С NER и радикальным режимом:**

```bash
docker run -d -p 6003:6003 \
  -e ANONYMIZATION_ENABLED=true \
  -e ANONYMIZATION_NER_ENABLED=true \
  -e ANONYMIZATION_RADICAL_MODE=true \
```

**Полная конфигурация через docker-compose.yml:**

```yaml
services:
  1c-mcp-proxy:

      - "6003:6003"
    environment:
      - ANONYMIZATION_ENABLED=true
      - ANONYMIZATION_TOOLS=execute_query,execute_code,get_object_by_link,get_event_log,find_references_to_object

      - ANONYMIZATION_NER_ENABLED=false
      - ANONYMIZATION_RADICAL_MODE=false
      - ANONYMIZATION_INCLUDE_ERRORS=true
```

## Сравнение режимов

| Аспект | Встроенный сервер | Прокси |
|--------|-------------------|--------|
| Где выполняется | В 1С (клиентский код BSL) | Python-прокси |
| Настройка | Форма в обработке 1С, дерево полей | Переменные окружения |
| Хранение настроек | ХранилищеОбщихНастроек 1С | Переменные окружения / память |
| Словарь | Запрос к БД при запуске сервера (и при включении анонимизации на уже запущенном сервере) | Асинхронная загрузка при первом запросе |
| Анализ имён полей | Да | Да |
| Регулярные выражения | Да (настраиваемые) | Да (встроенные) |
| SpaCy NER | Нет | Да (требует установленной модели SpaCy в окружении) |
| Умная анонимизация псевдонимов | Да (через схему 1С) | Да (через source hints схемы) |
| Изоляция по каналам | Единый контекст на базу | Отдельный маппинг на каждый канал |
| Де-токенизация параметров | Да | Да |

## Ограничения

- Маппинг токенов хранится **в памяти** и сбрасывается при перезапуске сервера/обработки.
- **SpaCy NER** доступен только в режиме прокси. Требует установленной модели SpaCy в окружении. В официальном Docker-образе предустановлена `ru_core_news_md`; при запуске вне Docker нужно установить нужную модель вручную (`python -m spacy download <model>`).
- `ANONYMIZATION_RADICAL_MODE=true` может чрезмерно токенизировать технические строки (имена процедур, метаданные). Рекомендуется использовать осторожно.
